Add a `materialized_table` materialization built on Flink's `CREATE OR ALTER MATERIALIZED TABLE ... AS SELECT`. It is declarative: every run re-asserts the definition and lets Flink reconcile it — a new table is created, any change (columns, `WITH` options, or query logic) is evolved in place, and an unchanged definition is a cheap no-op. `--full-refresh` drops (`DROP MATERIALIZED TABLE`) and recreates, which is also how you change `distributed_by` (fixed at creation). Supports `config(distributed_by={...}, with={...}, start_mode=...)`; `freshness_interval`, `refresh_mode`, and `partition_by` are rejected with a compile error. A re-run within Flink's brief establishment window is transiently rejected (`being modified`) and retried automatically. See MATERIALIZATIONS.md.
